OAuth

Secure Release Printing

Open Authorization (OAuth) is enhanced security that requires the Printer Installer client to use an "authorization code" to communicate with Printer Installer. The authorization codes are something you set up and manage in the Admin Console. Any Printer Installer client will be able to communicate with Printer Installer so long as the authentication code it is using has not expired. Multiple clients can use the same code.

To enable OAuth, you must first update Printer Installer to version 18.1. For more information on updating to 18.1 see your PrinterLogic Support Engineer.

There are three ways in which a client can be validated with the server.

  • If the end-user has administrator access to the Admin Console, then they are recognized as having an acceptable level of access and will be authorized.
  • Configuring the authorization by Group Policy Object (GPO) or deployment product.
  • Authorize end-users from the Self-service Portal. When an end-user clicks the link to install a new printer, a window will appear, requesting the authorization code.

Install OAuth through either the standard or an advanced 18.1 Printer Installer installation, or through the Printer Installer 18.1 upgrade. If you are not sure whether you have OAuth enabled or not in Printer Installer, just click the Tools menu. If the Authorized Devices setting appears, then OAuth is enabled on Printer Installer, as shown below.

If you are currently using Printer Installer 15.1, you will need to update to Printer Installer 16.1 or 17.1 first and then upgrade again to 18.1 to start using OAuth.

OAuth in the Standard Install

A fresh Printer Installer 18.1 standard installation automatically enables OAuth on the Printer Installer server. As such, you will not see any OAuth settings specifically in the standard installation wizard, but be assured that OAuth is included in the installation, which means that the 18.1 clients deployed to the workstations must use an authorization code to communicate with Printer Installer. For more information, see Install Printer Installer.

OAuth in the Advanced Install

The Printer Installer advanced installation, like the standard installation, automatically enables OAuth on the Printer Installer server as part of the 18.1 installation. However, unlike the standard installation, the advanced installation provides settings for your Printer Installer clients, as shown in the image to the left.

Because 18.1 is a fresh install with OAuth enabled on the Printer Installer server, you must indicate whether you want to immediately force the clients to connect to Printer Installer, to postpone this requirements until a specific date, or to not use the authentication codes at all.

For example, let's say you select Always Enabled in the advanced installation wizard. Moreover, you already have Printer Installer clients installed in a test environment. If those Printer Installer clients are not upgraded to 18.1 then they will no longer be able to connect to the Printer Installer server, the reason being that OAuth on the Printer Installer server now requires all Printer Installer clients to have an authentication code.

Because of the immediacy and overall impact to your environment that the Always Enabled field can have on your environment, make sure you understand what you are doing when you select Always enabled as any clients that are 17.1 or earlier will no longer function with 18.1. For more information, see Install Printer Installer.

If you are not ready for the clients to use authentication codes just yet, click Enabled on and then select a date. By postponing client authorization to a later date, you are providing yourself with time to update the clients, but for the meantime, Printer Installer will not require the Printer Installer clients to use authentication codes. When the date you select arrives, however, you need to have your authentication codes set up and have updated the clients on your workstations to Printer Installer 18.1 so they can use the authentication codes to communicate with Printer Installer. If you have no interest in authenticating your Printer Installer clients through OAuth, you can select Always disabled. OAuth will still be enabled on the Printer Installer server because you are installing 18.1, but the clients will never need to use an authentication code to communicate with Printer Installer.

OAuth in the 18.1 Upgrade

The 18.1 upgrade can have immediate and lasting effect on your existing printing environment. For this reason, the upgrade is not available without assistance from a PrinterLogic Support Engineer. The 18.1 upgrade provides settings for enabling OAuth on the Printer Installer server and for configuring the existing clients to communicate with Printer Installer. For more information, see Upgrade an Installation.

The 18.1 upgrade wizard provides OAuth configuration settings for both Printer Installer server and client. If you are upgrading to 18.1, then you already have a Printer Installer environment set up, and enabling OAuth can invalidate the Printer Installer clients on your workstations - unless you have replaced the existing clients on your workstation with 18.1 clients or configured the system to postpone the use of 18.1 clients.

The image to the left shows the OAuth configuration settings in the 18.1 upgrade wizard. The top section, OAuth Support, is for enabling OAuth on your Printer Installer server. You will notice the section does not contain an option for disabling OAuth on the server. If you select Always enabled and then finish the upgrade, OAuth will be enabled on your Printer Installer server. If you are not ready just yet to enable OAuth on your Printer Installer server, select Enable on and then select a date.

The settings in the Require Client Authorization section determine exactly how Printer Installer reacts to the Printer Installer clients. For example, let's say you select Always enabled from the OAuth Support section, and then you select Always enabled from the Require Client Authorization section. When the upgrade is finished installing, any Printer Installer 17.1 or earlier clients installed on your workstations will no longer work because the Printer Installer server now requires the Printer Installer clients to use an authorization code, which you also set up in Tools > Authorization Codes. If you select Always enabled for both OAuth sections in the upgrade wizard, then you should be ready to deploy the 18.1 clients to prevent any type of disruption.

While the Always enabled option represents a quick upgrade for users who don't have a large number of clients to redeploy, the Enable on option is better for most users because you can postpone enabling OAuth on the Printer Installer server and postpone the use of authentication codes to communicate with the server. For example, let's say that in both OAuth settings sections, you select Enable on. After you finish the upgrade, Printer Installer will continue to function as it has previously, but when the selected date arrives, Printer Installer 18.1 will require authentication codes from the Printer Installer clients, which must be updated to 18.1 as well.

If you would like to upgrade to 18.1, but you are not interested in using OAuth, just select Always enabled in the OAuth Support section and then select Always disabled in the Require Client Authorization section as shown below.

With these settings, OAuth will still be enabled on the Printer Installer server, but the clients will not use any authentication codes to communicate with Printer Installer.

Printer Installer 18.1 uses OAuth to enhance security through the use of "tokens" called authentication codes, which are used to verify the client as a Printer Installer client. The authentication codes are created in the Admin Console, but there are multiple ways in which they can be applied to an 18.1 client. However, before you can set up your authentication codes, you must enable OAuth on the server through either the 18.1 fresh install or upgrade. Also, if you are enabling OAuth on your Printer Installer server, and you plan on using OAuth, then you must update the clients on your workstations to 18.1, which can be done via client deployment, end-user authorization in the Self-service Portal, or auto-authorization via the Admin Console.

After you have successfully installed or upgraded to Printer Installer 18.1, and your installation or upgrade has been configured to enable OAuth, you will see the Authorized Devices setting in the Tools menu as shown in the image here.

If you do not see the Authorized Devices setting, then the use of OAuth was postponed during the installation. The Authorized Devices menu item enables you to create the authorization codes your Printer Installer clients will be using. Printer Installer will be requiring two things specifically - an 18.1 or later client and an authentication code.

The authentication codes can work in a number of ways. You can apply a single code to multiple workstations, which can be useful when you are testing. You can also create multiple codes for multiple workstations, or even a one code to one client configuration.

Each code is authorized for a specific period of time. Expiration lengths include 30 minutes, 60 minutes, 2 hours, 24 hour, 48 hours, and No Expiration. If you want the code to be used quickly and for a short-term, then you will select the shorter time period. If it does not need to be used right away, you can set it for a longer time period.

After the code is generated, it is displayed in the Active Authorization Codes section, where you can see the actual code, the expiration length, and the date and time the code will expire. However, the code is not considered "in use" until it appears in the Active Devices section as well, which requires you to assign the code to a client. If the code is not used by the date and time in the Expiration Date column, then it will be removed from the list of codes in the Active Authorization Codes section.

When a code is applied to a client, it displays in the Active Devices section. Expired Authorization codes will continue to display in the Active Devices section until you manually remove them by clicking the Garbage icon.

Step 1: Generate an Authentication Code

  1. In the Admin Console, click Tools > Authorized Devices.
  2. In the Expiration Length field, select the amount of time the code will be valid before it expires.

NOTE: A code can be used to install the client on several machines. It is not necessarily a one code to one machine configuration, but you can set up a one to one if you would like. For this reason, the code is highly useful for installing the client using a Group Policy Object (GPO). For more information, see Install Printer Installer Client.

  1. Click Generate.
  1. The new code will appear in the Active Authorization Codes section.

Step 2: Authenticate the Printer Installer Client

Authorization codes enable the client on an end-user's workstation to securely communicate with the PrinterCloud server. The first two options below apply the code to the client, while the third process automatically authorizes the client by virtue of their administrator access to the Admin Console.

Option 1: Deploy the Authorization Code Automatically

Use the following command to securely authorize the client(s) on one or more workstations:

msiexec /i PrinterInstallerClient.msi HOMEURL=http://PrinterInstallerPortalURL REINSTALLMODE=vomus REBOOT=ReallySupress REINSTALL=ALL AUTHORIZATION_CODE=add the generated code here /norestart

Option 2: End-user Authorization

End-user authorization occurs from the Self-service Portal when the end-user attempts to install a printer. In this case, the following image appears, requesting the code from the administrator. By entering the correct code, a token will be placed on the end-user's workstation, and the name of the workstation will appear in the Admin Console, specifically in the Active Devices section. When the administrator removes the code from the Active Authorization Codes field, then the client will no longer be able to connect to the server.

 

Option 3: Auto-authorize via Admin Console Access

After a device has been authorized with a code, the name of the device will appear in the Generate Authorization Code window under the Active Devices section as shown here.

Any workstations connected to the device will remain in the list and be connected to the server until you remove them manually. In other words, even if the code expires, the workstations connected to the PrinterCloud server will remain connected until you remove the workstation name from the Active Devices list. Furthermore, after the code expires, no additional workstations can use the code to connect to the sever.
